# DataHub Semantic Search

Система интеллектуального поиска данных по табличным файлам и документам на естественном языке.

Проект предназначен для автоматического анализа файлов, генерации их текстовых описаний с помощью LLM, сохранения метаданных и семантических представлений в PostgreSQL, а затем поиска подходящих источников данных по запросу пользователя через RAG-подход.

## Описание

`DataHub Semantic Search` – это прототип хранилища данных с умным поиском, в котором каждый файл предварительно анализируется, описывается LLM и индексируется для последующего поиска по смыслу.

Основной фокус проекта – **табличные данные**: Excel, CSV и похожие структурированные источники.  
Идея проекта – не просто хранить файлы, а превращать их в **поисковый каталог данных**, в котором можно найти нужный файл или таблицу по запросу на естественном языке.

## Ключевая идея

Обычный поиск по имени файла или по ключевым словам плохо работает для больших и разнородных наборов данных.  
Вместо этого каждый файл проходит этап анализа:

1. Извлекается структура файла.
2. Определяются листы, колонки, типы данных, объём, примеры строк.
3. LLM генерирует человекочитаемое описание содержания файла.
4. Описание преобразуется в embedding.
5. Embedding и метаданные сохраняются в PostgreSQL.
6. Пользователь задаёт вопрос на естественном языке.
7. Система находит наиболее релевантные файлы по семантической близости.

Таким образом, проект работает как **интеллектуальный data catalog**, а в будущем может быть (возможно) расширен до полноценного **data hub** с поиском и использованием самих данных файлов.

## Возможности

- Загрузка и индексирование табличных файлов.
- Поддержка метаданных: путь, имя файла, размер, тип, дата загрузки, автор (тот, кто загрузил).
- Извлечение структуры таблиц: листы, колонки, количество строк.
- Генерация текстового описания файла через LLM.
- Семантический поиск по смыслу запроса.
- Хранение embeddings в PostgreSQL через `pgvector`.
- Поддержка фильтрации по типу файла, имени, дате и другим метаданным.
- Подготовка архитектуры для будущего перехода к запросам по данным, а не только по каталогу.

## Архитектура

Проект состоит из нескольких этапов обработки:

### 1. Ingestion step
Файл загружается в систему и сохраняется в файловом хранилище или на диске.

### 2. Parsing step
Система определяет формат файла и извлекает структуру:
- для Excel – листы, заголовки, примеры строк;
- для CSV – колонки, статистика, sample rows;
- для документов – текстовое содержимое и метаданные.

### 3. LLM application step
LLM генерирует описание:
- что это за файл;
- какие данные он содержит;
- для чего может использоваться;
- какие есть ключевые сущности и поля.

### 4. Indexing step
Текст описания преобразуется в embedding и сохраняется в PostgreSQL.

### 5. Retrieval step
Пользователь вводит запрос на естественном языке, система строит embedding запроса и ищет наиболее близкие записи в БД.

### 6. Answering layer
Система возвращает:
- найденные файлы;
- при необходимости – краткое объяснение, почему они подходят;
- при необходимости – дополнительные метаданные.

## Как это работает

Пример сценария:

Пользователь пишет:

> Покажи таблицы с продажами по регионам за 2023 год

Система:
1. Преобразует запрос в embedding.
2. Находит в каталоге описания, похожие по смыслу.
3. Возвращает, например:
   - `sales_2023.xlsx`
   - `regional_revenue_q4.csv`
4. Показывает описание таблицы, путь к файлу и релевантные метаданные.

## Структура данных

Ниже пример таблицы, в которой хранится каталог файлов:

```sql
CREATE TABLE files_catalog (
    id BIGSERIAL PRIMARY KEY,
    file_name TEXT NOT NULL,
    file_path TEXT NOT NULL,
    file_type TEXT NOT NULL,
    sheet_names TEXT[],
    columns JSONB,
    row_count BIGINT,
    file_size BIGINT,
    metadata JSONB,
    description TEXT NOT NULL,
    embedding VECTOR(1536),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

### Смысл полей

- `file_name` — имя файла.
- `file_path` — путь до файла.
- `file_type` — формат файла: xlsx, csv, pdf и т.д.
- `sheet_names` — список листов, если это Excel.
- `columns` — структура колонок.
- `row_count` — количество строк.
- `metadata` — дополнительные метаданные.
- `description` — текстовое описание, созданное LLM.
- `embedding` — векторное представление описания.

## Принцип генерации описания

Для каждого табличного файла система может использовать:
- список листов;
- названия колонок;
- первые несколько строк;
- статистику по количеству строк;
- типы данных;
- пропуски;
- уникальные значения в некоторых колонках.

На основе этого LLM формирует компактное, но информативное описание, например:

> Таблица содержит данные о продажах за 2023 год. Основные поля: дата, регион, товар, количество, сумма. Есть разбиение по месяцам и филиалам. Подходит для анализа выручки и региональной динамики.

Именно это описание затем индексируется для семантического поиска.

## Ограничения

- Проект не предназначен для прямой обработки миллионов строк в одном embedding.
- Это не замена DWH, а интеллектуальный каталог и слой поиска поверх данных.
- Для очень больших таблиц лучше индексировать не все строки, а схему, агрегаты и sample rows.
- LLM может ошибаться в интерпретации содержимого, если структура файла плохо оформлена.
- Для точного анализа данных в будущем лучше дополнить систему Text-to-SQL.

## Планы развития

- Поддержка более сложных форматов файлов.
- Индексация не только описаний, но и отдельных листов.
- Гибридный поиск: embeddings + полнотекстовый поиск.
- Добавление reranking.
- Интеграция Text-to-SQL для перехода от поиска файлов к запросам по данным.
- Построение полноценного data hub.
- Разделение на уровни: каталог, витрины, аналитика, поиск, генерация ответов.
